# Stage 8 - flattener and submissions

Run metadata: `prosperity4btest==5.0.0`, all round-days per round (round 1:
-2/-1/0, round 2: -1/0/1, round 3: 0/1/2, round 4: 1/2/3, round 5: 2/3/4),
generated by `p4alpha.flatten.flatten` from current `src/p4alpha/` source.

## 1. What was built

`src/p4alpha/flatten/flatten.py`: an AST-driven flattener that concatenates
each `strategies/roundN.py` with its transitive `core/` dependency closure
into a single competition-legal file.

- **Dependency resolution is real, not hard-coded.** `core_dependencies()`
  reads a module's own top-level imports; `transitive_core_closure()` BFS's
  outward (so a strategy importing `core.ou` would correctly also pull in
  `core.indicators`, ou.py's own dependency, even though no strategy does
  this today); `topological_order()` is Kahn's algorithm with an
  alphabetical tie-break for a deterministic, reproducible order. Confirmed
  directly (`tests/flatten/test_flatten.py`): round 1/2/5 need
  `{execution, fair_value, indicators}`; round 3/4 additionally need
  `options`; `core.ou` is never pulled in by any of the five (no strategy
  imports it).
- **Text reproduction is line-range deletion, not `ast.unparse`.** Only
  three things are deleted from each source module, by AST-derived line
  range: its own leading module docstring, `from __future__ import
  annotations` (one combined copy is hoisted to the top of the output,
  since Python requires future-imports to be contiguous at the very start
  of a file), and internal `p4alpha.core*` imports (dead after
  concatenation - the names are already in scope in the same file).
  Everything else survives verbatim, **comments included**: a comment is
  never an AST node, so `ast.unparse` would have silently discarded every
  `#` decision-note comment beside a strategy constant, and this project's
  params-block rule (CLAUDE.md) is that those comments carry the research
  citation for the number. Function/method docstrings also survive (they
  are never the module's own leading docstring), so e.g.
  `submissions/round1_submission.py` still carries `_trade_ash`'s own
  docstring explaining its `tiers`/`extreme_threshold` override kwargs.
- **Soundness is checked, not assumed.** `_assert_no_nested_internal_imports`
  rejects (rather than mishandles) a hypothetical `p4alpha.core` import
  nested inside a function/class, since line-range deletion cannot safely
  strip those the way it strips top-level ones. `_assert_no_name_collisions`
  rejects two concatenated modules defining the same top-level name (a
  silent-shadowing bug; a no-op today, since no two of the five core
  modules ever collide, but cheap and real). The assembled source must
  both `ast.parse` and `compile()` (a stricter syntax check - e.g. catches
  `return` outside a function); `compile()` only validates syntax, the
  resulting code object is never executed, per PLAN.md §7's "the flattener
  manipulates AST and never executes strategy code". `check_banned_imports`
  confirms every surviving top-level import in the output is stdlib or
  `datamodel` (the competition's runtime-injected module) - never
  `p4alpha` itself, since the entire point is a submission that runs with
  no `p4alpha` install at all.
- **`submissions/round{1..5}_submission.py`**, generated and committed, one
  per round, plus a `--check` mode (`python -m p4alpha.flatten.flatten
  --check`) that regenerates in memory and fails loudly if the committed
  files have drifted from current source (PLAN.md §6: "regenerated by the
  flattener and diff-checked in CI").
- **`tests/flatten/test_flatten.py`** (54 tests): dependency
  resolution/topological order (including a synthetic circular-dependency
  case via a monkeypatched `CORE_DIR`, and a hash-seed-stability test
  spawning real subprocesses under different `PYTHONHASHSEED` values -
  gate review item 1), line-range stripping on real core/ and strategy
  files, name-collision and banned-import unit tests, the
  submissions-freshness/regeneration test (same byte-for-byte pattern as
  `research/regime.py`'s Stage 4 precedent), and a per-file syntax +
  banned-import test against the five *committed* submission files
  directly (PLAN.md's literal DoD wording).
- **`tests/flatten/test_parity.py`** (5 tests, one per round, marked
  `parity`): backtests both the source strategy and its flattened
  submission through the real engine on every day that round covers, and
  asserts the two activity logs are byte-identical (strictly stronger than
  a PnL-only check - it would also catch a position/fill divergence that
  happened to net to the same final number).

## 4. Carried-forward limitations (item 3): unchanged, not silently altered

Flattening is line-range deletion of dead imports/docstrings around
otherwise-verbatim source; it cannot change runtime behaviour, and section
2's byte-identical activity logs are the direct, measured proof of that,
not an assumption:

- **Round 3/4 correlation-stacking exposure cap** bounds new-order risk,
  not a static book's continuous mark-to-market drift as vouchers' deltas
  change (Stage 5/6 finding, docs/results/round3/backtest.md,
  docs/results/round4/backtest.md). Unchanged by flattening; still an open
  item for a future stage (a per-instrument cap generalised to a
  continuously-rebalanced one).
- **Round 5 shared order-book volume template** across several
  `GBM_OUTER_PRODUCTS` concentrates correlated fill risk beyond what a
  signal-differentiated strategy would have (Stage 7 finding,
  docs/results/round5/backtest.md section 4-5). Unchanged by flattening;
  still an open item (a per-template rather than per-product exposure
  cap).

Neither limitation is a flattener concern to fix - both are strategy-level
findings this stage's own DoD explicitly carries forward unresolved.

## 6. Security note (PLAN.md §7)

No new secret, endpoint, or externally-reachable input point. `flatten.py`
reads only repo-local paths it derives itself (`CORE_DIR`,
`STRATEGIES_DIR`, computed from `__file__`), never a user-supplied path;
its only "parsing of untrusted input" is `ast.parse` over this repo's own
already-reviewed source files. No `eval`/`exec` anywhere: `compile()` is
used exactly once, purely to validate syntax, and the resulting code
object is discarded, never executed - the literal requirement in PLAN.md
§7 ("the flattener manipulates AST and never executes strategy code"). No
new third-party dependency: `flatten.py` is stdlib-only (`argparse`, `ast`,
`sys`, `dataclasses`, `pathlib`), consistent with `core/`'s own constraint
even though `flatten/` isn't literally bound by that enforced test.
`submissions/*.py` are generated build artefacts, not hand-edited, and
`check_banned_imports` plus a dedicated test against the five *committed*
files confirm none of them import anything beyond stdlib and the
competition's own runtime-injected `datamodel` module.

## 8. Gate review (pre-commit), three items resolved

1. **Topological ordering determinism, confirmed, not assumed.**
   `topological_order`'s Kahn's-algorithm tie-break already picks
   `sorted(...)[0]` (alphabetically first ready module) at every step, so
   Python's per-process `set` hash-randomisation is fully normalised away
   before it can affect output order - verified concretely, not just
   reasoned about: `flatten_strategy()` for all five rounds was run in
   five separate fresh interpreter processes with `PYTHONHASHSEED=random`,
   and again with `PYTHONHASHSEED` pinned to `0`/`1`/`42`; the full
   generated source (all five rounds, sha256) was byte-identical every
   time. Now permanently guarded by
   `test_flatten_output_is_stable_across_hash_seeds`
   (`tests/flatten/test_flatten.py`), which spawns real subprocesses with
   different `PYTHONHASHSEED` values rather than asserting in-process,
   since the whole risk is process-to-process hash randomisation. No code
   change was needed; the existing tie-break was already correct.
2. **`--check` now runs as its own explicit CI step**, not only implicitly
   via a pytest test's internal regenerate-then-compare mechanics.
   `.github/workflows/ci.yml`'s `flatten-parity` job gained `uv run python
   -m p4alpha.flatten.flatten --check` as a standalone step between the
   sync and the pytest invocation: a stale or hand-edited committed
   submission now fails the job immediately, with its own dedicated error
   message, before any test (parity or otherwise) runs, independent of
   pytest's test-execution order.
3. **Generated-file header: commit hash evaluated and omitted, plain
   header kept.** The header already built in this stage's first pass
   (`_header()`, `src/p4alpha/flatten/flatten.py`) already opens with
   `# Generated by p4alpha.flatten.flatten -- DO NOT EDIT BY HAND.` as its
   first line - matching the requested marker, and predating this review
   round rather than being newly added by it. A commit-hash addition was
   evaluated concretely against this repo's own git state: a hash embedded
   at generation time can only ever be the *parent* of the commit that
   ships the file (a commit's hash is a function of its own tree, which
   cannot already contain the finished file's own not-yet-computed hash),
   so the first `--check` run against the commit that actually ships the
   file would see a different `HEAD` and fail *permanently*, not
   intermittently - self-invalidating by construction, not merely a risk.
   Per the explicit fallback instruction, the hash is omitted; the
   reasoning is recorded in a code comment directly above `_header()` as
   well as here. Regenerated and re-verified after this review round
   (ruff clean, 542 non-parity tests + 5 parity tests, all green) before
   the closure commit.
